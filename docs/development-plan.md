1. `Live` 서버 시간 모드 오류 분석 및 안정화
2. 아이템 카탈로그 정리
3. 메일 시스템 부활
4. 스태미나 시스템 개선
5. 가챠 테이블 UI 개선
6. 미션 시스템 복원
7. 멀티플레이 안정화 및 공유 초대 복원
8. 멀티플레이 매칭 타임아웃 및 봇 채우기 검토
9. 배포용 레포/문서/업데이트 흐름 정리

## 현재 기준

- 게임 클라이언트는 수정하지 않는다. APK, SWF, IL2CPP, 클라이언트 바이너리는 명시적인 이유가 생기기 전까지 건드리지 않는다.
- 가능한 한 모든 기능은 서버 측 구현으로 해결한다.
- 멀티플레이는 현재 방 생성/입장, 준비 상태, 배틀 시작, 행동 동기화, 일부 보상, 룸 복귀/해산까지 동작한 기록이 있다.
- 관리자 UI에는 Dashboard, Players, Rooms, Items, Source Code 탭이 있다.
- 직접 아이템 지급은 현재 `free_vmoney`, `free_mana`를 지원한다.
- `TZ` 기본값은 한국 커뮤니티 운영을 전제로 `Asia/Seoul`이다.

## 증거 없이 하지 말 것

- UI 상태만 보고 멀티 프로토콜을 다시 작성하지 않는다.
- 메일, 스태미나, 미션 응답 구조를 추측으로 만들지 않는다.
- `.database`, `.generated`, `.cdn`, 플레이어 상태를 기능 개발 중 임의로 삭제하거나 재생성하지 않는다.
- 로그, CDN 파일, 키, 에뮬레이터 산출물, 개인 세이브 데이터를 커밋하지 않는다.
- 클라이언트에 fallback AI가 있는지 확인하기 전까지 서버 측 봇 AI를 직접 구현하지 않는다.

## 1. `Live` 서버 시간 모드 오류 분석 및 안정화

현재 상태:

- `Fixed` 모드는 정상 운영 기준으로 사용 가능하다.
- `Live` 모드는 클라이언트 로그인/로드 중 약 5% 지점에서 `No.H500` 실패를 일으킨다.
- 원인이 확인되기 전까지 일반 테스트와 운영은 `Fixed` 모드로 진행한다.

목표:

- `Fixed`: 게임 서버 시간을 선택한 특정 시각에 고정한다. 기본값으로 유지한다.
- `Live`: 선택한 가챠 테이블의 날짜에 서버의 현재 `TZ` 기준 시각을 결합한다.
- 향후 스태미나, 일일 미션, 메일 발송 시간 처리는 `fixed`/`live` 상태에 따라 분기할 수 있어야 한다.

다음 조사:

- `Fixed` 성공 흐름과 `Live` 실패 흐름의 HTTP API 로그를 비교한다.
- `.database/server-time.json`, `/api/server/timeState`, API 응답의 `servertime` 값을 확인한다.
- `No.H500` 직전에 어떤 엔드포인트가 실패하는지 특정한다.
- 선택된 가챠 날짜와 현재 시간이 가챠, 상점, 캠페인, 튜토리얼, 로그인 상태가 기대하는 기간을 벗어나는지 확인한다.

주의:

- H500 원인이 확인되기 전까지 스태미나, 일일 미션, 메일 발송 시간 기능을 `Live` 모드 위에 얹지 않는다.

## 2. 아이템 카탈로그 정리

현재 상태:

- 관리자 UI에서 `free_vmoney`, `free_mana` 지급은 가능하다.
- 실제 운영을 위해서는 아이템 ID, 이름, 아이콘, 카테고리, 지급 가능 여부가 정리되어야 한다.

목표:

- 에셋과 CDN 데이터에서 로컬 아이템 카탈로그를 만든다.
- 관리자 UI에서 아이템 이름과 아이콘으로 검색할 수 있게 한다.
- 이 카탈로그를 메일 첨부, 보상 지급, 이벤트 지급의 공통 기반으로 사용한다.

필요한 근거:

- 압축 해제된 CDN 파일 구조.
- 아이템 이름과 아이콘 리소스를 참조하는 asset JSON.
- 대표 세이브 파일 안의 보유 아이템/재화 구조.

## 3. 메일 시스템 부활

현재 상태:

- 게임 내 메일 탭은 열리지만 서버 측 메일 시스템은 구현되어 있지 않다.
- 현재 아이템 지급은 세이브 데이터에 직접 반영되는 방식이다.

목표:

- 관리자가 텍스트 메일, 아이템 메일, 재화 메일을 보낼 수 있게 한다.
- 대상은 단일 유저, 선택한 여러 유저, 전체 유저를 지원한다.
- 유저는 게임 내 기존 메일 UI에서 메일 확인, 단일 수령, 일괄 수령을 할 수 있어야 한다.
- 공지사항에 가까운 텍스트 메일도 보낼 수 있으면 좋다.

필요한 근거:

- 메일 탭 진입 시 HTTP 로그.
- 단일 수령/일괄 수령 시 HTTP 로그.
- 빈 우편함과 메일이 있는 우편함의 응답 구조.

## 4. 스태미나 시스템 개선

현재 상태:

- 스태미나는 사실상 고정 또는 무한에 가깝게 동작한다.
- UI상으로 회복되는 것처럼 보이지만 실제 서버 상태와 일관되지 않을 수 있다.

목표:

- 관리자 설정으로 `무한 스태미나`와 `정상 스태미나`를 선택할 수 있게 한다.
- 정상 스태미나 모드는 퀘스트 소모량, 자연 회복, 회복 아이템, 랭크별 최대 스태미나, 랭크업 시 전체 회복을 지원한다.

의존성:

- 서버 시간 모드가 안정화되어야 한다.
- 스태미나 회복 아이템 ID와 아이템 사용 API를 확인해야 한다.

## 5. 가챠 테이블 UI 개선

현재 상태:

- 관리자 UI의 가챠 테이블 선택은 ID와 날짜 범위만 표시한다.
- 운영자가 어떤 배너인지 직관적으로 알기 어렵다.

목표:

- 가챠 이름, 배너 이미지, 픽업 캐릭터/장비, 적용 기간을 표시한다.
- 날짜가 겹치는 테이블이 있을 수 있으므로 날짜만 기준으로 판단하지 않는다.
- 알 수 없는 테이블을 위해 수동 선택 경로를 유지한다.

필요한 근거:

- `assets/gacha.json`
- `assets/gacha_campaign.json`
- CDN 배너/리소스 매핑.
- 필요하면 실제 클라이언트 화면 스크린샷.

## 6. 미션 시스템 복원

현재 상태:

- 미션 탭은 열리지만 활성 미션이 표시되지 않는다.

목표:

- 우선 간단한 일일 미션부터 복원한다.
- 기존 미션 UI에서 진행도 확인과 보상 수령이 가능해야 한다.
- 관리자가 일일 미션 활성화 여부와 보상을 설정할 수 있게 한다.

의존성:

- 서버 시간 모드.
- 아이템 카탈로그.
- 보상 지급 공통 처리. 가능하면 메일 시스템과 연결한다.

## 7. 멀티플레이 안정화 및 공유 초대 복원

현재 상태:

- 멀티플레이의 큰 흐름은 동작하지만, 연결 해제, 전멸, 포기, 방 해산, 보상 지급 등은 계속 정교화가 필요하다.
- 실제 테스트에서 연결이 끊긴 파티가 남은 클라이언트에서 봇처럼 움직이는 듯한 현상이 있었다.
- 원래 게임에는 방장이 공유를 누르면 다른 유저 화면 왼쪽 위에 종 표시 알림이 뜨고, 그 알림을 누르면 해당 방 정보를 확인한 뒤 참가할 수 있는 기능이 있었다.

목표:

- 방 생성/입장/해산/복귀/이탈/전멸/포기 처리의 상태 정합성을 안정화한다.
- 아무도 없는 방이나 끊긴 세션이 Rooms에 남지 않게 한다.
- 보상 지급이 중복되거나 과다 지급되지 않게 한다.
- 공유 버튼을 통한 초대 알림을 복원한다.
- 공유 알림에서 방 정보를 확인하고 참가할 수 있게 한다.
- 번호를 외부 메신저로 따로 공유하지 않아도 같은 서버의 유저가 방을 찾을 수 있게 한다.

공유 초대 조사 대상:

- 방장이 공유 버튼을 눌렀을 때 호출되는 HTTP API와 요청/응답 구조.
- 다른 유저가 종 알림을 받기 위해 주기적으로 호출하는 API가 있는지 확인한다.
- 공유 알림을 눌렀을 때 방 정보 조회와 참가가 어떤 엔드포인트로 이어지는지 확인한다.
- 공유 알림에는 방 번호만 있는지, 방장/퀘스트/참가자 정보까지 들어가는지 확인한다.
- 기존 `search_room`, `get_rooms`, `share_room` 구현과 겹치는 부분을 정리한다.

우선순위 판단:

- 공유 초대 복원은 실제 유저 편의에 직접 영향을 주므로 멀티 안정화 항목 안에서 중간 우선순위로 둔다.
- 단, 방 생성/입장/해산/복귀/보상 정합성이 먼저 안정적이어야 한다.

## 8. 멀티플레이 매칭 타임아웃 및 봇 채우기 검토

현재 상태:

- 예전 구현 과정에서 방이 자동으로 해산되거나 끊기는 문제를 피하기 위해 일부 타임아웃 동작을 제거하거나 완화한 적이 있다.
- 원래 게임은 멀티방에서 매칭 후 일정 시간이 지나면 부족한 자리를 봇이 채우는 듯한 동작이 있었다.
- 실제 테스트에서도 연결이 끊긴 파티가 남은 클라이언트에서 봇처럼 움직이는 듯한 현상이 있었다.

봇 채우기 가설:

- 원래 서버가 매칭 타임아웃 후 가짜 참가자 또는 봇 참가자를 넣었을 수 있다.
- 또는 서버가 특정 disconnect/fallback 메시지만 보내고, 실제 봇 동작은 클라이언트가 로컬에서 처리했을 수 있다.
- 우리가 제거하거나 완화한 타임아웃 중 일부가 원래는 봇 채우기나 자동 시작과 연결되어 있었을 수 있다.

나중에 할 실험:

- 원래 클라이언트가 기대하는 매칭 타임아웃 값과 타임아웃 표시 UI를 확인한다.
- 타임아웃이 되었을 때 클라이언트가 보내는 요청과 서버가 보내야 하는 메시지를 확인한다.
- 디버그 브랜치에서 방 매칭 타임아웃을 통제해서 재현한다.
- 인원이 부족한 방에 가짜 참가자를 추가한다.
- 배틀 시작 후 가짜 참가자를 접속시키지 않거나 즉시 disconnect/bye 패턴을 보낸다.
- 남은 클라이언트가 해당 파티를 로컬 봇으로 전환하는지 확인한다.

우선순위 판단:

- 봇 채우기는 있으면 좋은 기능이지만, 실시간 멀티 안정화와 보상/이탈 처리보다 후순위다.
- 먼저 타임아웃 로직이 어떤 목적이었는지 확인하고, 이후 클라이언트 fallback AI를 활용할 수 있는지 검증한다.

## 9. 배포용 레포/문서/업데이트 흐름 정리

현재 상태:

- Docker 기반 배포는 가능하다.
- 공개 배포를 위해서는 불필요한 파일, 로컬 산출물, 테스트 흔적이 남지 않도록 계속 관리해야 한다.

목표:

- 공개 레포에는 필요한 소스, 배포 파일, 문서, 템플릿만 남긴다.
- CDN 데이터, 개인 키, 세이브, 로그, 에뮬레이터 산출물, 임시 분석 결과는 포함하지 않는다.
- README는 실제 사용법 위주로 유지한다.
  - clone
  - `.cdn` 생성
  - CDN 데이터 배치
  - `.env` 설정
  - `docker compose up -d`
  - 관리자 로그인
  - WireGuard QR 생성
  - 백업, 업데이트, 롤백
- 비상업 팬 프로젝트라는 안내를 명확히 적는다.
- `git pull`과 `docker compose up -d --build`가 `.cdn`, `.database`, `.generated`를 삭제하지 않도록 유지한다.
