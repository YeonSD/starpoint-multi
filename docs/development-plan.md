1. 서버 시간 모드 안정화
2. 메일 시스템 부활
3. 아이템/재화 카탈로그 정리
4. 가챠 테이블 표시 개선
5. 멀티플레이 초대/공유 기능 복원
6. 멀티플레이 매칭 타임아웃 및 봇 채우기 조사

## 현재 기준

- 게임 클라이언트 자체는 수정하지 않는다. APK, SWF, IL2CPP 바이너리는 명확한 근거가 생기기 전까지 서버 분석용 참고 자료로만 사용한다.
- 기능 구현은 가능한 한 Starpoint 서버, realtime 서버, WireGuard/DNS/nginx 구성으로 해결한다.
- 멀티플레이는 방 생성, 입장, 준비, 전투 시작, 실시간 동기화, 결과 보상, 룸 복귀까지 주요 흐름이 동작한다.
- 관리자 UI에는 Dashboard, Players, Rooms, Items, Source Code 탭이 있다.
- Items 탭은 메일 지급을 기본값으로 사용한다. 현재 즉시 필요한 재화는 `free_vmoney`, `free_mana`, `exp_pool`이다.
- 직접 지급은 테스트/운영 보정용으로 유지한다. 음수 지급은 직접 지급에서만 지원하고, 메일 지급은 양수 보상만 허용한다.
- `TZ` 기본값은 한국 커뮤니티 운영 기준으로 `Asia/Seoul`을 사용한다.

## 증거 없이 하지 말 것

- UI 상태만 보고 프로토콜을 추측해서 구현하지 않는다.
- 메일, 스태미나, 미션 응답 구조를 실제 HTTP 로그나 pcode 근거 없이 새로 만들지 않는다.
- `.database`, `.generated`, `.cdn`, 플레이어 저장 데이터를 임의로 삭제하거나 재생성하지 않는다.
- 로그, CDN 파일, 키 파일, 개인 세이브 데이터는 공개 레포에 커밋하지 않는다.

## 1. 서버 시간 모드 안정화

현재 상태:

- `Fixed` 모드는 선택한 시각에 서버 시간을 고정한다. 기본 운영 모드로 유지한다.
- `Live` 모드는 선택한 날짜와 실제 서버 타임존의 현재 시각을 결합한다.
- `Live` 모드는 메일 발송 시각, 일일 미션 리셋, 향후 시간 기반 운영 기능을 검증하기 위한 기반이다.
- 스태미나 정식 회복 시스템은 보류한다. 현재는 기존 save data 기반의 사실상 무한 스태미나 흐름을 유지한다.

목표:

- Dashboard에서 현재 서버 시간이 명확히 보이도록 한다.
- `Fixed`/`Live` 선택 상태가 서버 재시작 후에도 유지되어야 한다.
- 메일 표시 시간은 UTC 오차 없이 서버 타임존 기준으로 표시되어야 한다.

## 2. 메일 시스템 부활

현재 상태:

- `/mail/index`, `/mail/receive`, `/mail/receive_all` 기본 경로가 구현되어 있다.
- 미수령 메일의 `receive_time`은 클라이언트가 null로 해석하는 `"0000-00-00 00:00:00"`을 사용한다.
- 메일 응답의 `type`은 `MailKind`가 아니라 클라이언트 `MailType` wire 값이다.
- 현재 확인된 재화 메일 타입:
  - `free_vmoney`: `type=4`, `type_id=null`, `number=수량`
  - `free_mana`: `type=8`, `type_id=null`, `number=수량`
  - `exp_pool`: `type=9`, `type_id=null`, `number=수량`
- 성도석 메일은 실제 수령 후 `free_vmoney`가 증가하는 것을 확인했다.

목표:

- 관리자 UI에서 특정 유저, 선택 유저, 전체 유저에게 메일을 보낼 수 있게 한다.
- 메일 제목, 본문, 보상 종류, 수량을 운영자가 지정할 수 있게 한다.
- 메일 수령 시 실제 save data에 보상이 반영되어야 한다.
- 일반 아이템 메일은 아이템 카탈로그 정리 후 `type/type_id/number` 구조를 확장한다.

후순위:

- 게임 내 “수령 이력” 메뉴가 현재 H404로 실패한다. 보상 수령 자체와는 별개이므로 후순위로 복원한다.
- 텍스트만 있는 공지 메일, 만료 기간이 있는 메일, 일괄 수령 상세 동작을 추가로 검증한다.

## 3. 아이템/재화 카탈로그 정리

현재 상태:

- `free_vmoney`, `free_mana`, `exp_pool`은 아이템 ID가 아니라 플레이어 재화 필드 및 메일 타입으로 처리한다.
- 일반 아이템은 `players_items`와 CDN/asset의 아이템 마스터를 연결해야 한다.

목표:

- CDN과 asset 데이터에서 아이템 ID, 이름, 아이콘, 카테고리를 추출한다.
- 관리자 UI에서 이름과 아이콘으로 검색해서 메일에 첨부할 수 있게 한다.
- 일반 아이템, 장비, 캐릭터, 각종 포인트류를 단계적으로 확장한다.

---

## 4. 가챠 테이블 표시 개선

현재 상태:

- Dashboard에서 가챠 테이블을 선택할 수 있지만 ID와 기간만 보여 운영자가 의미를 알기 어렵다.

목표:

- 가챠 이름, 배너 이미지, 픽업 캐릭터/장비, 적용 기간을 표시한다.
- 실제 운영 날짜와 Starpoint 내부 날짜가 다를 수 있으므로, 날짜만으로 의미를 단정하지 않는다.

## 5. 멀티플레이 초대/공유 기능 복원

조사할 기능:

- 방에서 공유를 누르면 다른 유저 화면 왼쪽 위에 종 알림이 뜨고, 그 알림을 누르면 해당 방 정보로 참가할 수 있는 기능이 있다.
- 룸 번호를 별도로 전달하지 않아도 입장 가능하지만, 누르기 전에는 방 구성원을 알기 어렵다.

목표:

- `attention` 관련 API와 멀티방 공개/공유 요청을 분석한다.
- 현재 방 리스트 및 공유 상태를 기반으로 알림이 전달되는지 확인한다.
- 실제 클라이언트가 기대하는 초대 payload를 복원한다.

---

## 6. 멀티플레이 매칭 타임아웃 및 봇 채우기 조사

조사할 기능:

- 원래 멀티방에서 매칭 후 일정 시간이 지나면 봇이 빈자리를 채워 전투가 시작되는 흐름이 있었던 것으로 보인다.
- 현재 구현에서는 안정화를 위해 일부 timeout 동작을 제거하거나 완화했다.
- 전투 중 유저가 끊기면 상대 파티가 봇처럼 움직이는 현상이 관찰되므로, 클라이언트에 AI 대체 로직이 존재할 가능성이 높다.

목표:

- 매칭 timeout 관련 서버 설정과 클라이언트 상태 전이를 확인한다.
- 빈 슬롯을 봇으로 채우는 동작이 서버 메시지인지, 클라이언트 내부 fallback인지 구분한다.
- 우선순위는 낮지만, 장기적으로 1인 멀티방 자동 출발 기능 후보로 둔다.

---

## 7. 스태미나 시스템 개선

현재 상태:

- 현재 운영은 save data의 스태미나 값을 계속 서버가 내려주는 방식이다.
- 클라이언트 UI상 자연 회복은 5분당 1로 보이지만, 서버 정식 회복/소모/아이템 사용 처리는 아직 구현하지 않는다.

목표:

- 기본값은 기존 무한 스태미나 흐름을 유지한다.
- 향후 선택 옵션으로 정상 스태미나 모드를 구현할 수 있다.
- 정상 모드 구현 시 랭크별 최대 스태미나, 5분당 1 회복, 회복 아이템, 성도석 회복, 랭크업 회복을 모두 검증해야 한다.

목표:

- 일일 미션부터 우선 복원한다.
- 관리자가 미션 사용 여부, 조건, 보상을 설정할 수 있게 하는 것을 장기 목표로 둔다.
- 이벤트 미션과 일반 미션은 후순위다.

## 9. 수령 이력 및 운영 로그 정리

목표:

- 게임 내 수령 이력 메뉴를 복원한다.
- 관리 페이지에서 메일 발송, 직접 지급, 정기 지급, 수령 여부를 확인할 수 있게 한다.
- 운영자가 실수로 같은 보상을 여러 번 지급했는지 추적할 수 있게 한다.

---

## 10. 배포 패키지, 업데이트 절차, 공개용 문서 정리

목표:

- 처음 사용하는 운영자가 `git clone`, `.cdn` 배치, `docker compose up -d`만으로 서버를 띄울 수 있어야 한다.
- `.cdn`, `.database`, `.generated`, `.logs`, `key`는 공개 레포에 포함하지 않는다.
- 업데이트는 기존 DB와 WireGuard 설정을 보존하면서 `git pull` 및 재배포가 가능해야 한다.
- README는 사용법 중심으로 작성한다.
- 원본 Starpoint 프로젝트에 대한 출처와 존중을 명확히 표시한다.
- 비상업적 팬 프로젝트이며 상업적 이용을 금지한다는 문구를 포함한다.
