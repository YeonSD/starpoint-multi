# 아이템 카탈로그 분석 계획

## 원칙

- 게임 클라이언트는 변조하지 않는다.
- APK, SWF, CDN, 저장 데이터는 분석 자료로만 사용한다.
- 실제 구현은 Starpoint 서버, 관리자 페이지, 서버 DB/캐시에만 적용한다.
- 아이템 지급 기능은 검증된 타입만 기본 노출하고, 미검증 항목은 별도 표시한다.

## 현재 확인된 사실

### 재화와 일반 아이템은 분리해야 한다

메일 지급에서 확인된 재화 타입은 다음과 같다.

| 항목 | Save Data 필드 | 메일 type | type_id |
| --- | --- | ---: | --- |
| 성도석 | `free_vmoney` | 4 | `null` |
| 마나 | `free_mana` | 8 | `null` |
| 캐릭터 경험치 풀 | `exp_pool` | 9 | `null` |

반면 플레이어가 보유하는 일반 아이템은 DB의 `players_items` 및 응답의 `item_list`에 `id -> amount` 형태로 저장된다.

따라서 카탈로그 구조는 최소한 아래처럼 분리해야 한다.

- `currency`: 성도석, 마나, 경험치 풀처럼 플레이어 필드로 관리되는 항목
- `item`: `item_list` / `players_items`에 들어가는 일반 아이템
- 이후 필요 시 `equipment`, `character`, `bond_token` 등 별도 타입 추가

### 일반 아이템 ID 후보는 이미 상당수 확보 가능하다

다음 파일들에서 보상 `type: 0` 또는 `itemId`를 기준으로 일반 아이템 ID 후보를 추출할 수 있다.

- `assets/treasure_shop.json`
- `assets/box_gacha.json`
- `assets/box_reward.json`
- `assets/score_reward.json`
- `assets/rare_score_reward.json`
- `assets/event_item_shop.json`
- `assets/boss_coin_shop.json`
- `assets/clear_reward.json`
- `assets/save_data.json`

현재 간단 추출 기준으로 약 400개 이상의 일반 아이템 ID 후보가 나온다.

예시:

- `1` ~ `16`
- `42` ~ `67`
- `99`, `101`, `102`
- `10001`, `10002`, `10003`
- `11001`, `11002`, `11003`
- `12001` 계열
- `14001` 계열
- `30000` 계열
- `40000` 계열
- `49000` 계열
- `100000`, `500000`, `980000`, `990000`, `999000` 계열

이 목록은 “ID 후보”일 뿐이며, 이름/아이콘/메일 지급 가능 여부는 별도 검증이 필요하다.

### CDN 아이콘 추출은 가능성이 높다

`.cdn/ko/entities/2.1.121-android_medium.csv`는 다음 형태의 엔티티 매니페스트다.

```text
path,version,size,sha256,group
```

CDN zip 내부 파일명은 논리 이름이 아니라 해시 경로다.

예:

```text
production/upload/00/0d25178949b68815791e143896af9b5c8afa95
```

샘플 파일을 확인한 결과, 일부 이미지 파일은 표준 PNG 시그니처가 아니라 아래처럼 들어 있다.

```text
89 70 6e 67 0d 0a 1a 0a ...
```

표준 PNG는 `89 50 4e 47`이므로, `png` 부분이 소문자다. 이 3바이트를 대문자로 고치면 이미지로 열 수 있다.

확인된 샘플:

- 캐릭터/썸네일 계열 이미지
- 작은 아이템 아이콘 계열 이미지
- UI 부품 이미지

즉 아이콘 파일 자체는 CDN에서 추출할 수 있다. 문제는 아직 `아이템 ID -> 해시 이미지 파일` 매핑이 확정되지 않았다는 점이다.

### 클라이언트 코드상 썸네일 타입도 분리되어 있다

분석된 클라이언트 코드에서 다음 타입이 확인된다.

- `pinball.ui.component.item.ItemThumbnailKind`
- `ItemThumbnailKind.Item(param1:String, param2:int)`
- `ItemThumbnailKind.Stone`
- `ItemThumbnailKind.Mana`
- `ItemThumbnailKind.PooledExp`
- `ItemThumbnailKind.StarCrumb`
- `ItemThumbnailKind.BondToken`
- `ItemThumbnailKind.Equipment(...)`

이 구조상 성도석/마나/경험치 풀은 일반 아이템 ID 하나로 처리하지 않고 별도 썸네일 타입으로 처리된다.

## 아직 확정되지 않은 것

### 아이템 이름의 출처

현재 `assets/*.json`에는 `name: ""`처럼 비어 있는 값이 많다.

따라서 한국어/영어 이름은 아래 중 하나에서 찾아야 한다.

- CDN 내부 텍스트 리소스
- SWF/AS3 리소스 문자열
- 로컬라이즈 테이블
- 클라이언트가 런타임에 로드하는 별도 마스터 데이터
- 기존 문서화된 route 응답 샘플

### 아이콘 매핑 규칙

아이콘 파일은 찾을 수 있지만 파일명이 해시라서 직접 매칭이 안 된다.

가능한 경로:

1. 클라이언트 코드에서 `ItemThumbnailKind.Item(...)`를 만드는 지점을 찾아 item id/group 값을 추적한다.
2. CDN 내부의 스프라이트/텍스처 메타데이터를 찾아 논리 이름과 해시 파일을 매칭한다.
3. 게임 내 아이템 화면을 정상 API 응답과 비교해서 ID, 이름, 아이콘을 수동 검증한다.
4. 관리 UI에는 우선 신뢰도 필드를 두고, 확정된 항목부터 노출한다.

## 추천 구현 구조

카탈로그 엔트리는 아래 형태가 적당하다.

```ts
type CatalogKind = "currency" | "item" | "equipment" | "character";

interface ItemCatalogEntry {
  key: string;
  kind: CatalogKind;
  id: number | null;
  saveField?: "free_vmoney" | "free_mana" | "exp_pool";
  mailType: number;
  mailTypeId: number | null;
  nameKo: string;
  nameEn: string;
  descriptionKo?: string;
  descriptionEn?: string;
  iconPath?: string;
  sources: string[];
  confidence: "confirmed" | "inferred" | "unknown";
}
```

초기에는 다음만 `confirmed`로 둔다.

- 성도석
- 마나
- 캐릭터 경험치 풀

일반 아이템은 다음 순서로 확정한다.

1. `item_list`에 존재하고 보상 테이블에도 등장하는 ID
2. 메일 지급 `type=1`, `type_id=itemId`가 실제 수령되는지 검증
3. 이름과 아이콘까지 매칭된 항목

## 사용자 협조가 있으면 빨라지는 검증

다음 동작은 클라이언트 변조가 아니며, 정상 게임 조작과 화면/로그 확인만 필요하다.

1. 게임 내 `메뉴 -> 아이템` 화면에서 여러 아이템이 보이는 상태를 캡처한다.
2. 특정 아이템 수량이 있는 계정의 save JSON을 내려받아 `item_list`와 화면을 비교한다.
3. 서버에서 특정 일반 아이템 ID를 메일로 1개 보내고, 메일함 진입/수령/수량 증가 여부를 확인한다.
4. 아이템 화면에서 같은 아이템의 아이콘과 한국어 이름을 캡처한다.
5. 필요하면 영어 클라이언트 환경에서 같은 아이템 화면을 캡처해 영문 이름을 매칭한다.

## 다음 작업 순서

1. CDN pseudo-PNG 인덱서 작성
   - zip을 풀지 않고 이미지 후보의 경로, 크기, 차원을 색인화한다.
   - `.generated` 아래에 캐시한다.
2. 일반 아이템 ID 후보 추출기 작성
   - `assets/*.json`과 `assets/save_data.json`에서 ID 후보와 출처를 수집한다.
3. 이름 테이블 탐색
   - `.analysis`의 pcode/AS3 문자열과 CDN 텍스트 리소스에서 item name 계열 키를 찾는다.
4. 카탈로그 초안 생성
   - 이름/아이콘이 없는 항목도 ID와 출처만으로 `unknown` 상태로 포함한다.
5. 관리 UI 적용
   - 검색 가능 select로 교체한다.
   - 확정 항목 우선 표시, 미확정 항목은 경고 표시.
6. 일반 아이템 메일 지급 검증
   - 검증된 ID부터 메일 지급 옵션에 추가한다.
